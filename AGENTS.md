# AGENTS.md

## 适用范围

本文件适用于仓库根目录及其所有子目录。若子目录后续新增更具体的 `AGENTS.md`，以距离目标文件最近的规则为准。

## 项目定位

本仓库用于学习和实践 [CloudWeGo Eino](https://github.com/cloudwego/eino)，重点沉淀可运行示例、实验代码和学习笔记，不作为 Eino 上游源码的镜像或替代文档。

当前仓库仍处于初始化阶段：仅有项目说明，尚未创建 Go 模块、源码目录和自动化测试。执行任务时必须以仓库中的实际文件为依据，不得假定某个目录、依赖版本或命令已经存在。

## 工作原则

1. 修改前先阅读 `README.md`、`go.mod`、目标目录文档和相关测试；文件不存在时不要虚构其约定。
2. 优先完成单一、可验证的学习目标，保持改动小且聚焦，避免顺手重构无关代码。
3. Eino API、行为和版本信息以上游仓库及官方文档为准；无法从当前依赖版本验证时，明确标注假设。
4. 示例应能够独立理解和运行。新增示例时，同时说明用途、前置条件、运行命令和预期结果。
5. 默认使用简体中文编写仓库文档、注释说明和示例说明；代码标识符遵循 Go 社区惯例。
6. 不修改用户本地 IDE 配置、未跟踪文件或与当前任务无关的内容。

## 目录约定

仓库形成稳定结构前按需创建目录，不为占位而新增空目录。推荐使用以下布局：

- `examples/<主题>/`：可独立运行的 Eino 学习示例，每个主题保持边界清晰。
- `notes/`：概念、源码阅读和实验结论等学习笔记。
- `internal/`：仅在多个示例确实需要共享实现时使用，不要过早抽象。
- 与包同目录的 `*_test.go`：单元测试和行为测试。

若实际代码形成了不同且更合理的结构，应更新本文件和 `README.md`，而不是机械套用上述目录。

## Go 与 Eino 编码规范

- 以 `go.mod` 声明的 Go 版本和依赖版本为准；新增依赖后保持 `go.mod` 与 `go.sum` 同步。
- 所有 Go 代码必须通过 `gofmt`。包名简短、全小写，导出标识符按 Go 规范补充必要注释。
- 需要取消、超时或链路传递的操作使用 `context.Context`，并将其作为首个参数向下传递。
- 错误必须被处理或显式返回；补充上下文时使用 `%w` 保留错误链，禁止仅记录后继续假装成功。
- 优先使用 Eino 提供的公开 API 和组件接口，不复制上游内部实现，不依赖 `internal` 包或未公开行为。
- 每个示例聚焦一个核心主题。除非学习目标就是集成多个组件，否则避免把模型、模板、工具、检索和持久化全部堆入同一示例。
- 外部模型、数据库和网络服务通过配置或环境变量接入。密钥、令牌、真实账号和生产地址不得写入代码、测试、日志或提交文件。
- 环境变量示例使用无敏感信息的占位值；新增必需配置时同步说明变量名、用途和是否必填。
- 单元测试默认不得访问真实外部服务；通过接口、桩或受控测试实现隔离。需要在线调用的实验必须显式标记，并避免纳入默认测试路径。

## 文档要求

新增或修改示例时，相关说明至少包含：

1. 学习目标和涉及的 Eino 组件。
2. 依赖的 Go/Eino 版本及外部服务前置条件。
3. 完整、可直接执行的命令。
4. 关键输入、预期输出和已知限制。

学习结论应区分“源码或运行结果已验证”和“待验证推断”。引用上游行为时尽量附对应官方文档、源码文件或版本标签。

## 验证要求

当前尚无 `go.mod`，因此没有可执行的仓库级 Go 构建或测试命令。初始化 Go 模块后，提交代码前至少执行：

```bash
gofmt -w <本次修改的 Go 文件>
go test ./...
go vet ./...
```

验证遵循以下规则：

- 优先运行与改动直接相关的最小测试，再运行仓库级测试。
- 新增行为应补充正常路径、错误路径和关键边界条件测试。
- 若因缺少凭据、网络、外部服务或工具而无法执行验证，必须说明未执行项、原因和剩余风险，不得声称已经通过。
- 纯文档改动至少检查链接、命令、路径和描述是否与仓库当前状态一致。

## 变更边界

- 不提交密钥、生成物、缓存、IDE 工作区文件或本机绝对路径。
- 不随意升级 Go、Eino 或其他依赖；确需升级时说明动机、兼容性影响和验证结果。
- 不为通过测试而删除断言、跳过测试或掩盖错误。
- 不执行破坏性 Git 操作，不覆盖用户未提交的改动。
- 新增工具链、代码生成或自动化脚本时，必须同时提供可复现命令和必要文档。
